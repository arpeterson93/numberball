## What already exists: the scouting-recency stoplight (pitcher-only, sequence-pattern-only)

`pages/2_Scouting.py`'s pitcher "Swing Suggestions" panel has an Inspector tab (`_stoplight_inspector`, `pages/2_Scouting.py:275+`) showing a green/yellow/red state per signal, backed entirely by `utils.py`:

No batter identity or rating data enters this at all.
- `_surprisal_walk(context_keys, outcome_buckets, k, alpha=1.0)` (`utils.py:3763-3790`) walks pitches chronologically; at each pitch, using ONLY counts accumulated from *prior* pitches (point-in-time, add-alpha smoothed), it predicts a distribution over `k` outcome buckets, scores the actual bucket via `_score_from_probs` (`ln(k·p_obs)`, `utils.py:3751-3760`), then updates the running counts. - `scouting_recency_states`/`scouting_recency_detail` (`utils.py:3678+`, `3774+`) are signal-agnostic wrappers around the above, driving the aggregate light and the Inspector's per-pitch drill-down respectively.

## What's different about the "OBP recent X" rows

Three Suggestions rows - "OBP recent pitch", "OBP recent Δ", "OBP recent Δ²" - each build a candidate population (raw recent pitches / `project_from_deltas` output / `project_from_delta2s` output, respectively - see `utils.py`), then call `utils.obp_zone_signal(pop, ranges, weights, maximize, paired=...)` (`utils.py`, recently added/reworked), which finds `best_val` (the swing value maximizing expected on-base probability via FFT convolution of the population against `ranges` - `_scores_via_fft`/`_diff_score_array`, `utils.py:2142-2157`) and grows a zone around it capped at the matchup's OBR width.

`ranges` (`active_ranges` for the pitcher tab) is matchup-specific in the live UI: it comes from `_calc_ranges()` (`pages/2_Scouting.py:790-833`), pulling the CURRENTLY selected pitcher's and batter's rating attributes into `utils.compute_at_bat_ranges(...)` (`utils.py:5030-5061`). A fully "honest" backtest would need to reconstruct this per historical at-bat (historical opposing batter's ratings, that at-bat's situational flags, etc.) - **this is explicitly ruled out below.**

## Decisions already made (do not re-litigate - plan around these)

**Decision A - reuse the CURRENT live zone geometry for every historical point, not a per-at-bat reconstruction.** Do NOT look up historical opposing batters' ratings, do NOT reconstruct `result_ranges` per historical row, and do NOT call `compute_at_bat_ranges` inside the backtest loop at all. Instead: take the zone WIDTH that the CURRENTLY selected live matchup already produces for that indication right now (i.e. whatever `obp_zone_signal` already computed for "OBP recent pitch"/"Δ"/"Δ²" this run - the `left`/`right` it grew, or equivalently `hi - lo`), and reuse that same width for every historical point in the backtest. This sidesteps the entire historical-ratings-lookup problem (no per-row batter/situational reconstruction, no data-quality risk, no extra lookups) at the cost of answering a slightly different question: "how has this pitcher's own pattern performed against a zone shaped like TODAY's recommendation," not "against whatever the true historical matchup was at the time." That tradeoff is accepted - do not re-open it.

Still needed per historical point: **where** the zone is centered. Recompute `best_val` at each historical index the normal way (FFT-score the point-in-time-windowed population - see Decision C - against a `ranges`; use the SAME current live `active_ranges`/`result_ranges` for this too, not a historical reconstruction, for the same reason as the width). The zone's *center* moves with the pitcher's evolving recent pattern; its *width* is pinned to today's live value throughout.

**Decision B - the backtest is NOT a binary hit/miss test. It's a categorical bucket-prediction backtest with unequal-width buckets**, restoring compatibility with the existing `_surprisal_walk`/`_aggregate_recency` shape rather than inventing a separate binary mechanic. Bucket design:
- **Bucket 0 ("recommended")**: the current OBR-capped zone around `best_val`, width `W` (from Decision A). Its position moves per historical index (tracks `best_val`); its width is fixed at today's value throughout.
- **Buckets 1..K-1 ("other")**: the remaining `1000 - W` of the domain, divided into `K-1` buckets of equal width to each other (NOT equal to `W`). **Bucket count is decided, not a free parameter** - size the "other" buckets as close to `W` itself as possible: `remaining = 1000 - W`; `n_other = max(1, round(remaining / W))`; then divide `remaining` into `n_other` buckets as evenly as integer division allows, distributing any leftover one unit at a time (`base, rem = divmod(remaining, n_other)`; the first `rem` buckets get `base+1`, the rest get `base`). Worked example: `W=328` -> `remaining=672` -> `672/328=2.049` -> rounds to `n_other=2` -> `672` splits evenly into `[336, 336]`. Another: `W=270` -> `remaining=730` -> `730/270=2.704` -> rounds to `n_other=3` -> `divmod(730,3)=(243,1)` -> `[244,243,243]`. Edge cases to guard: `W>=1000` (no remainder, degenerate - only the recommended bucket exists, this signal has nothing to score against); very small `W` producing a large `n_other` (e.g. `W=1` -> 999 one-wide buckets) - Fable should propose a sane floor/ceiling (e.g. a minimum "other" bucket width, or a maximum `n_other`) rather than let this run unbounded, and flag what a reasonable real-world `W` range looks like empirically before picking that floor/ceiling.
- **Per-bucket baseline probability is WIDTH-PROPORTIONAL, not uniform**: `p0[bucket] = width[bucket] / 1000`, not `1/k`. This is what makes unequal-width buckets valid - the "no tendency" reference for a bucket has to reflect how much of the domain it actually covers.
- **Generalize `_score_from_probs`**: replace `ln(k · p_obs)` with `ln(p_obs / p0[observed])`. This is a strict generalization - it collapses to the exact original formula when every bucket happens to be equal width (`p0 = 1/k` for all), so the existing signals' behavior is unaffected if this generalized formula is used everywhere (recommend doing so for one shared implementation rather than a fork).
- **The "recommended" bucket's ABSOLUTE position moves over time (it tracks `best_val`), but track counts by RELATIVE bucket index, not absolute location - this still works.** Index buckets ordinally relative to the recommended zone at each point in time: index 0 = the recommended zone wherever it currently sits, index +1 = the next equal-width "other" chunk going one direction around the circular wheel from it, +2 the next, and so on until the full 1000-wide domain is tiled at that instant. At each historical step, determine which RELATIVE index the actual observed value falls into given THAT step's own partition (anchored at that step's `best_val`), and accumulate a single running count array `counts[i]` over these relative indices across all historical steps - exactly the same mechanism `_surprisal_walk` already uses for its context→outcome counts, just with one constant/trivial "context" (no conditioning variable - this is a single always-on distribution, not a lookup table keyed by a separate context) and `p0[i] = width[i]/1000` per index instead of uniform `1/k`. This is a genuine multi-bucket categorical distribution over time, not a collapsed binary hit/miss - `P(index i)` is meaningfully trackable and reportable for every index, not just "recommended vs. not."
- **Two DISTINCT quantities, do not conflate them**: `p0[i]` is FIXED forever (pure geometry - what fraction of the domain index `i` covers), while the model's actual per-step prediction `p_pred[i]` is LEARNED from accumulated history and evolves over time, starting equal to `p0[i]` with no history and drifting away from it as real observations accumulate. `p_obs` (used in the score) is `p_pred[observed index]` at that step - not `p0[observed index]`. Confusing these two is an easy mistake to make when first reading this design - Fable should make the distinction explicit in code (e.g. `p0` and `p_pred`/`p_obs` as clearly separately-named values, never reusing one variable for both).
- **Corrected smoothing formula** (the natural first guess here is subtly wrong - flag this explicitly so Opus doesn't repeat the mistake): to exactly reduce to the existing `_surprisal_walk` formula when all buckets are equal-width (the backward-compatibility claim below depends on getting this right), pseudo-counts per index must be `alpha · k · p0[i]` (not `alpha · p0[i]`), with denominator `total + alpha · k` (not `total + alpha`):
  ```
  p_pred[i] = (counts[i] + alpha·k·p0[i]) / (total + alpha·k)
  ```
  Check: when `p0[i] = 1/k` for all `i`, this becomes `(counts[i] + alpha) / (total + alpha·k)` - the existing formula exactly. `k` here is the total bucket count (1 recommended + `K-1` other).
- **Generalized score at each step**: `ln(p_pred[observed index] / p0[observed index])`.

**Decision C - population windowing per historical index still applies.** The live "OBP recent X" rows use a `tail(n_pitches)`-windowed population as of now. The backtest needs the equivalent window as of each historical index (only pitches strictly before that index, tailed to the same `n_pitches`). Confirm `project_from_deltas`/`project_from_delta2s` behave sensibly on these smaller, sliding historical windows; at-bats too early in the history to form a full window simply have no backtest score (same as the existing stoplight's `None`-context handling).

**Decision D - the Inspector's "P(bucket)% over time" trend chart needs a unit change for this signal.** `scouting_recency_linechart` (`utils.py`) plots raw `p_obs` over time with a FIXED dotted `1/k` baseline and green/yellow/red bands converted from the score thresholds via `p = exp(cutoff)/k` - all implicitly assuming one constant baseline for the whole chart. With unequal-width buckets, `p0[observed index]` varies depending on which index got hit at each historical point (a recommended-bucket hit carries a different baseline than an "other"-bucket hit), so raw `p_obs` on a fixed Y-axis is not comparable across time and would misrepresent the signal. **Fix: change this signal's Y-axis to the ratio `p_obs / p0[observed]` (equivalently `exp(score)`)** - always centered on 1.0 at "no tendency," directly comparable across time regardless of which index was hit, with the baseline line becoming a clean fixed `1.0` and the classification bands becoming fixed `exp(scouting_min)`/`exp(anti_max)` cutoffs again. Decide whether this ratio-based Y-axis should be signal-specific (only for the new unequal-bucket signals) or whether it's worth applying uniformly to `scouting_recency_linechart` for every signal (it would be a no-op relabeling for the existing equal-width signals, since `p0` is constant for them - just double-check this doesn't change any existing displayed numbers for those signals, only the axis's framing).

**This is a display-layer change only - it does not touch the stoplight itself.** `p_pred`/`p_obs` isn't hidden or discarded - it's still the quantity the ratio is computed from, and should still surface in hover tooltips and the drill-down table (Q4) alongside `p0` and which bucket was hit, for anyone who wants the raw number. And the actual green/yellow/red light - `_classify_pp(score)` and `_aggregate_recency`'s windowed vote - keeps running on the exact same `score = ln(p_pred/p0)` regardless of how the trend chart chooses to draw it. Nothing about Decision D changes what drives the stoplight; it only changes what's comfortable to look at on the trend line.
