## Decisions (build to these, do not re-derive)

**Decision 1 - fixed bucket width per row, tied to an existing slider, not OBR-derived.**
- Pitch: width = the Hot Pitch Zone Matrix **Following** bucket-size slider (`hz_follow_bucket_p`) - not Initial.
- Delta: width = the existing delta-heatmap bucket-size slider (`dd_bucket_p`).
- Delta²: width = the existing delta²-heatmap bucket-size slider (`dd2_bucket_p`).
- These are all fixed, human-chosen values from small option lists (e.g. `[25,50,100,125,250,500]`) - never derived from OBR, never variable per historical step. This is why nearly all of the old plan's complexity (variable `W_t`, the displacement-histogram fix, per-step zone regrowth) is gone: the partition is now static in shape (though NOT static in position - see Decision 3), so plain equal-width categorical prediction (matching the existing sequence signals' mechanism) applies directly.

**Decision 2 - the walk-forward backtest tests REAL historical values, not the synthetic projected population.** This is the methodological fix motivating this whole redesign, not just a simplification:
- **Pitch row**: population for finding `best_val_t` = real recent pitches in the point-in-time window (Decision 4). Since Pitch's "recent" population already IS the real data, there's no separate "real vs. synthetic" population to reconcile for this row - `best_val_t` and the tested outcome both come from the same real pitch sequence.
- **Delta row**: `best_val_t` still comes from `project_from_deltas` applied to the point-in-time windowed population (a synthetic/projected candidate set, same as today's live row). Convert `best_val_t` to `implied_delta_t = circular_signed_delta(last_pitch_t, best_val_t)`. The bucket built from `implied_delta_t` (Decision 3) is then tested against the pitcher's REAL historical delta sequence (`pitch_circ_delta`, i.e. the same real data `seq2_delta_hint` already uses) - NOT against another synthetic projected population. This is the change that breaks the earlier self-referential tautology (best_val came from a synthetic population; the evidence testing it is now independent, real data).
- **Delta² row**: same idea one level further - `best_val_t` from `project_from_delta2s`, converted to `implied_delta_t` then `implied_delta2_t = ||implied_delta_t| - |last_delta_t||` (where `last_delta_t` is the REAL signed delta into the pitch immediately preceding step `t`). Bucket built from `implied_delta2_t`, tested against the pitcher's real historical `pitch_circ_delta2` sequence.

**Decision 3 - bucket partition: centered "recommended" bucket + bidirectional (or unidirectional for Pitch) "out±N" tiling, with boundary handling.**
- **Pitch (circular domain, 1-1000, no edges)**: center a bucket of the Decision-1 width on `best_val_t`; it wraps naturally at the 1/1000 seam, same as the existing Hot Zone Matrix already handles. Tile the rest of the wheel in one consistent rotational direction: `out+1, out+2, ...` (matches the existing convention, no special edge case).
- **Delta/Delta² (bounded domain, 0-500, real edges)**:
  - **The recommended bucket**: if centering the full Decision-1 width on the implied value would push past 0 or 500, **widen-and-shift, don't truncate** - keep the bucket at exactly the slider's full width, sliding it to hug the boundary instead of shrinking it. Example: `implied_delta2=25`, bucket width=100 -> naive `[-25,75]` -> shift -> final bucket `[0,100]` (full width, preserved).
  - **Tiling the remainder on each side**: fill FULL slider-width buckets outward from the recommended one (`out-1, out-2, ...` below; `out+1, out+2, ...` above); the single OUTERMOST bucket on EACH side absorbs whatever doesn't divide evenly (floor-full-buckets-then-one-leftover, NOT "distribute the remainder evenly across every outer bucket"). This contains all non-uniform-width buckets to at most 2 total (one per side) - never the recommended bucket, never an interior "out" bucket. Worked example (`implied_delta=100`, bucket width=100 -> recommended `[50,150]`): low remainder = 50 -> one bucket `out-1=[0,50)` (width 50, no full 100-wide bucket fits); high remainder = 350 -> `out+1=[150,250)`, `out+2=[250,350)`, `out+3=[350,450)` (each width 100) then `out+4=[450,500]` (leftover, width 50). Total buckets sum to exactly 500.
  - Bucket count is naturally bounded by the slider's fixed option list (worst case `bucket_size=25` -> on the order of 20 buckets total) - no artificial cap constant is needed (unlike the earlier plan's `OBP_MAX_OTHER_BUCKETS`, which existed to guard against a pathologically small derived width that can't occur here).

**Decision 4 - ties in "optimal" broken toward the nearest candidate.** When `best_val_t`'s FFT-argmax has an exact tie (currently silently resolved by `np.argmax` picking the lowest index), break toward whichever tied candidate is closest (circularly for Pitch, numerically for the implied delta/delta² space) to the current reference point (the actual last real pitch, or the actual last real delta/delta² as appropriate). Same rule for all three rows.

**Decision 5 - walk-forward timing.** Window `[t - n_window, t)` (strictly before `t`) builds the prediction/bucket at step `t`; the value actually scored is the real, realized pitch/delta/delta² AT `t`. The first `n_window` pitches of history are never evaluable (no full window yet); the pitcher's most recent real pitch is always the last evaluable step (there is nothing after it to score against). This is the same point-in-time-safe discipline the existing stoplight signals already use.

**Decision 6 - Relevance Weighting should be included, using each step's own preceding state.** `best_val_t`'s population (Decision 2) should be weighted via `compute_pa_weights`, using the CURRENT live slider values (recency/result/state + g1/g2/g3 - these are the same for every historical step, since they're live UI state, not historical) but THAT STEP's own preceding `obc`/`outs` as the "current state" reference (not today's live matchup's outs/OBC) - so the weighting is point-in-time honest to what was actually happening at that at-bat.

**Decision 7 - stoplight scoring is unaffected by the 1-2 non-standard-width edge buckets.** `_score_from_probs`'s generalized form (`ln(p_obs / p0[observed])`, already in the codebase) handles arbitrary per-bucket `p0` correctly regardless of whether a bucket is the standard slider width or one of the two boundary leftovers - no new scoring logic needed, no impact on classification thresholds or the windowed vote.

**Decision 8 - Inspector trend chart uses raw `P(bucket)%`, not a ratio axis.** Do NOT build the ratio-based Y-axis fix from the earlier plan (it solved a problem specific to variable-width buckets that no longer exists here) - not even as unused/dormant code. Use the exact existing `scouting_recency_linechart` behavior (raw probability, fixed `1/k`-equivalent baseline line calibrated to the STANDARD bucket width) for all three new signals. The rare historical point that lands in one of the two non-standard leftover buckets will read against a slightly-off reference line - that's an accepted, disclosed simplification (see Decision 9), not silently wrong.

**Decision 9 - hover tooltip always discloses the observed bucket's true width/`p0`.** Whatever detail-row/tooltip mechanism the Inspector uses for these signals should always show the specific bucket's width and `p0` for that historical point, not just for the two edge-case buckets - so a careful look at any point reveals its real baseline, even though the chart's fixed reference line is calibrated to the standard (non-edge) width.

**Decision 10 - retrofit the SAME widen-and-shift boundary fix into every existing delta/delta² distance-based match in the app, not just the new stoplight.** One consistent convention everywhere, rather than the new feature introducing a third variant alongside two different pre-existing ones. Apply Decision 3's exact algorithm (widen-and-shift the matched window so it keeps its full requested width, sliding to hug the boundary instead of shrinking) to:
- `seq2_delta_hint`/`seq3_delta_hint` (`utils.py`) - the "2-Δ seq"/"3-Δ seq" Suggestions rows' `centered` branches.
- `seq2_delta2_hint`/`seq3_delta2_hint` (`utils.py`) - the "2-Δ² seq"/"3-Δ² seq" Suggestions rows' `centered` branches.
- `sequence_matches` (`utils.py:2702-2772`) for `domain in ("delta","delta2")` - the Sequence Viewer's "Delta" and "Delta²" tabs, including the `prior_val2` ("Last 2 values") match on `_p1`.

This is a deliberate, decided behavior change to already-shipped code, not something to hedge on - boundary-adjacent z-scores, hint ranges, and Sequence Viewer match populations will shift for contexts near 0 or 500 (wider effective matching where the old code silently under-matched). Include an explicit before/after comparison in the validation plan (see "What to produce") rather than treating it as risk-free, but do not gate it behind a separate decision - it ships together with this plan.
