## ⚠️ Naming collision — read carefully before writing the plan

## The second-order delta (Δ²) — what already exists

`pitch_circ_delta2` (`utils.py:983`, computed `utils.py:1000-1005`) already exists on the pitcher-side dataframe: `df.loc[sw, "pitch_circ_delta2"] = sw_df2.groupby(gk_pit2, group_keys=False)["pitch_circ_delta"].apply(lambda g: g.abs().diff().abs())`. In words: Δ² is the unsigned change between consecutive `|Δ|` values — "how much did the pitcher's movement magnitude change from one adjustment to the next," grouped per game+pitcher, non-negative, bounded `[0, 500]` same as Δ. It's already surfaced elsewhere as "Avg |Delta^2|" in a trend chart (`utils.py:6682`, `pitcher_ma_figure`), so the "Δ²" name has precedent in this codebase. **Note that functions like `next_delta_vs_prior_delta_heatmap`, `delta_third_dist`, `seq2_delta_hint`, and `seq3_delta_hint` all recompute their delta column fresh from the raw value column** (documented reason: "Always recalculate deltas fresh to ensure proper grouping for filtered data" — these functions run against filtered/sliced views of `df_p`, e.g. by pitcher or game). Their Δ²-equivalents will need to recompute Δ² fresh too — i.e. a **two-stage** recompute (first the `|Δ|` groupby-diff, then the `|Δ²|` groupby-diff-of-that), not just a read of the precomputed `pitch_circ_delta2` column. By contrast, `sequence_matches()` for `domain="delta"` reads the precomputed column directly rather than recomputing — its Δ² tab can most likely do the same (read `pitch_circ_delta2` directly), since it operates on the full `df_p`, not a re-sliced copy — but confirm this against how `df_p` is scoped before assuming it's safe to skip a recompute.

3. **Full stoplight/Inspector integration.** "2-Δ² seq" and "3-Δ² seq" must be added to `_STOPLIGHT_ORDER`, `_recency_indications`, and `_recency_labelers`, exactly like "2-Δ seq"/"3-Δ seq" are today, so they get a green/yellow/red scouting-recency state and show up in the Inspector tab's per-signal detail view.

4. **Suggestions rows show compounded pitch-range arms.** The "2-Δ² seq"/"3-Δ² seq" Suggestions rows must show actual predicted next-pitch ranges (`lo`/`hi`/`lo2`/`hi2`), not just a bare predicted `|Δ²|` bucket. Since a Δ² prediction is one derivative further removed from the pitch value than a Δ prediction, this requires compounding - and the compounding **must mirror the established Swing Analyzer projection `project_from_delta2s` (`utils.py:2017-2033`)**, which keeps the pitcher's **signed** last delta and adjusts only its magnitude:
   - The canonical model: `L = circular_signed_delta(prev_pitch, last_pitch)` is the **signed** last delta (range -500..+500). Given a predicted `|Δ²|` bucket `[d2_lo, d2_hi]`, the next **signed** delta lands in *either* `[L + d2_lo, L + d2_hi]` (magnitude grows in the same direction) *or* `[L - d2_hi, L - d2_lo]` (magnitude shrinks). Δ² moves the magnitude of the same-direction delta (acceleration/deceleration); it never flips direction. This is exactly `project_from_delta2s`'s inner `last_val + (last_delta + sign*d2)`, generalized from a point to a bucket.
   - Each signed next-delta range becomes ONE circular pitch arc off the last pitch: `((last_pitch + nd - 1) % 1000) + 1`. No clipping and no unreachable-bucket case (the 1-1000 wheel is fully reachable), unlike an unsigned `|Δ|` formulation.
   - Net effect: exactly **2 candidate pitch-range arms per Δ² bucket** (the `+` and `-` branches, merged to 1 when `d2_lo == 0`) - NOT 4. Do NOT use an unsigned `Δ² = |current_delta - next_delta|` formulation that runs both magnitude arms through `delta_to_pitch_ranges`'s pos/neg split; that discards `L`'s sign and produces spurious opposite-direction arms the canonical projector never generates. **Design the exact new helper (suggested `delta2_to_pitch_ranges(last_pitch, last_delta_signed, d2_lo, d2_hi)`), how it composes with `merge_delta_ranges` on the Δ² magnitude buckets and with `_delta_row_p`'s `r1`/`r2`/`extra` unpacking, and surface all arms** (with only 2/bucket the clutter concern is moot).

5. **Independent bucket-size control for Δ².** Do not reuse `dd_bucket_p`. Add a separate bucket-size selector (e.g. `dd2_bucket_p`, same `[25,50,100,125,250,500]` option set, own `st.select_slider` key) that drives the new heatmap, the new third-value distribution chart, and the Δ²-based Suggestions/Inspector bucketing consistently. Thread this new bucket size through every function/cache-key call site that currently threads `dd_bkt` (`_load_pitcher_stoplights`, `_load_pitcher_stoplight_detail`, `_recency_indications`, `scouting_recency_states`, `scouting_recency_detail` — check each signature for whether it needs a new `dd2_bkt` parameter alongside the existing `dd_bkt`).
